# SPEC: задачи от пользователей с LLM-валидацией

## Цель
Любой зарегистрированный пользователь может прислать задачу для битв. Корректность проверяет LLM. Присланные задачи попадают в бои **без рейтинга** (карантин) и переходят в рейтинговый пул только после аппрува модератора.

## Не-цели
- Репутация авторов, снижение депозита проверенным, ревшара, UI апелляций, автокалибровка сложности — позже.
- Мультисудейный консенсус для валидации — модель одна, смысла нет.
- Полноценная экономика кредитов. В первом срезе депозит = счётчик квоты, не деньги.

## Главная угроза, которую обязана закрывать конструкция
**Автор задачи её знает.** Если он (или его второй аккаунт) встретит свою задачу в бою, он подготовлен заранее, а Elo начислится настоящий. LLM-валидация эту дыру не закрывает — она про качество текста.

Механизмы:
1. **Жёсткое исключение автора из матчмейкинга** — задача с `created_by_user_id = X` НИКОГДА не связывается с боем, где X является владельцем любого из двух агентов. Для автора задача «сгорела».
2. **Карантин** — присланная задача сначала играется только в боях без рейтинга. Elo не двигается ни у кого, поэтому подготовка автора ничего не приносит.
3. **Аппрув модератора** — переход в рейтинговый пул только руками, со статистикой карантина перед глазами.

Честно: механизм 1 обходится сговором двух РАЗНЫХ людей (автор шлёт задачу сообщнику вне платформы). Это принимается как остаточный риск первого среза; его снижает то, что рейтинговый пул набирается только после аппрува, а аномальный winrate виден в статистике карантина.

## Текущее состояние (проверено по коду)
- `battle_tasks`: `source CHECK IN ('generated','company')`, `status CHECK IN ('draft','ready','retired')`, есть `created_by_user_id`, `secret`, `difficulty`, `category`, `rubric` (массив), `use_count`, `last_used_at`, `org_id`.
- Единственная точка создания — `POST /battles/tasks/generate`, `Depends(get_admin_user)`, `source` жёстко `GENERATED`.
- Выбор задачи из пула фильтрует `t.secret = TRUE` в 6 местах `battle_repo.py`: `diagnose_challenge` (294), `create_challenge` (872), `list_task_pools` (1031), `admit_to_queue` (1687), `abort_pool_exhausted` (1835), `_mark_queued` (1975).
- Рейтинг решается в `battle_service._decide_rated_eligibility` (343) и `battle_runner.settle_battle` (~282).
- Детектор инъекций уже есть: `battle_judges.detect_injection`.
- Бюджет судейских вызовов: `battle_budget.reserve_call` (леджер, дневные квоты, глобальный кап, брейкер).

## Подход
Выбран конвейер **заявка → дешёвые фильтры → LLM-валидация → карантин → аппрув → рейтинговый пул**.

Альтернатива (сразу в общий пул с пост-модерацией) отклонена: битая или «своя» задача сразу влияет на реальный Elo, откатывать рейтинг дорого и подрывает доверие.

Направление зависимостей: `api → service → repository`. Валидатор — сервис, не лезет в HTTP-слой.

## Изменения схемы — миграция V70 (V66-V69 заморожены)
`db/migrations/V70__battle_user_tasks.sql`:
- Расширить `battle_task_source_enum`: добавить `'user'`.
- Расширить `battle_task_status_enum`: добавить `'pending_validation'`, `'quarantine'`, `'rejected'`.
- Новые колонки: `validation_verdict JSONB`, `validation_reason TEXT`, `approved_by_user_id UUID REFERENCES users(id) ON DELETE SET NULL`, `approved_at TIMESTAMPTZ`, `quarantine_battles INT NOT NULL DEFAULT 0`.
- `content_key` — **generated-колонка**: `GENERATED ALWAYS AS (regexp_replace(btrim(lower(prompt)), '\s+', ' ', 'g')) STORED`, а не плоская колонка с отдельным бэкфиллом. То же самое выражение уже стоит в `battle_repo` в трёх местах (`_CONTENT_KEY`/`_CONTENT_KEY_T`/`_CONTENT_KEY_C`); колонка, заполняемая приложением, разъехалась бы с ним при первом же INSERT-пути, который про неё забыл, и уникальный индекс молча перестал бы дедуплицировать. Generated-форма делает расхождение невозможным и заполняет существующие строки прямо в `ADD COLUMN`, так что отдельный UPDATE-бэкфилл не нужен. Выражение иммутабельно, что и требуется для `STORED`.
- CHECK: задача не от `generated` может иметь `status='ready'` только при `approved_by_user_id IS NOT NULL`. Это структурно запрещает попадание непроверенной пользовательской задачи в рейтинговый пул.
- **Уникальный частичный индекс по `content_key` — `WHERE source = 'user' AND status <> 'rejected'`**, то есть только по заявкам пользователей, а не по всем не-`rejected` строкам. Дубли содержимого среди admin-задач — это не дефект, а осознанно допущенное состояние: V67 считает `COUNT(DISTINCT content_key)`, выбирает через `DISTINCT ON (content_key)` и гасит на связывании все дубли-близнецы именно потому, что засев их порождает (в живом пуле лежат пять ready/secret/неиспользованных `generated`-строк с одним ключом). Глобальный уникальный индекс противоречил бы этому инварианту и заставил бы оператора удалять засеянные строки ради миграции. Дедуп нужен там, где он реально нужен, — на пути подачи, включая гонку двух одновременных заявок с одним текстом. Остаточный риск: заявка против уже существующего `generated`-близнеца индексом не блокируется; не-гоночный случай ловит дешёвый фильтр валидатора, а связывание всё равно не даст такой задаче рейтинг до аппрува.
- Индекс под очередь модератора: `(status, created_at DESC) WHERE status IN ('pending_validation','quarantine')`.

## Изменения выбора пула (`battle_repo.py`, 5 мест связывания)
- **Рейтинговый пул**: `status = 'ready'` (как сегодня) `AND secret = TRUE`.
- **Пул без рейтинга**: `status IN ('ready','quarantine')` `AND secret = TRUE`.
- Разделение пулов выражено предикатом `_bindable_task_status(<rated_eligible>)` и применяется в трёх местах, где задача действительно выбирается: `admit_to_queue`, `abort_pool_exhausted`, `_mark_queued`. `rated_eligible` заморожен на accept, то есть на момент связывания уже известен. Это и есть основной механизм: рейтинговый бой карантинную задачу физически не получает.
- Два места-гейта до вызова (`diagnose_challenge`, `create_challenge`) намеренно считают только `status='ready'` — на момент вызова рейтинговость ещё не решена, поэтому счётчик меряет более строгий (рейтинговый) пул и не обещает вызов, который связывание потом не выполнит.
- **Исключение автора — во всех 5 местах** (`_task_not_authored_by_fighters`): `AND (t.created_by_user_id IS NULL OR (t.created_by_user_id IS DISTINCT FROM :owner_a AND t.created_by_user_id IS DISTINCT FROM :owner_b))`. Именно `IS DISTINCT FROM`, а не `NOT IN`: у открытого вызова `agent_b_owner_snapshot` = NULL, и `x NOT IN (a, NULL)` даёт NULL, то есть молча вычищает весь пул.
- **`list_task_pools` — исключения автора НЕТ, и это осознанно.** Эндпоинт публичный и неаутентифицированный, сравнивать не с кем; протаскивать туда авторизацию значит менять контракт публичного эндпоинта ради косметики. Последствие зафиксировано: автор может увидеть в агрегате число, завышенное на его собственные задачи. Утечки нет — про свои заявки автор и так знает, — а связывание его задачи всё равно исключает.

## Изменение гейта рейтинга
- **Основной механизм — разделение пулов на связывании** (см. предыдущий раздел), а не проверка при расчёте. `_decide_rated_eligibility` вызывается на accept (`challenge_pending → accepted`), а задача связывается позже, на `reserved → queued`: на момент замораживания рейтингового вердикта задачи ещё физически нет, поэтому причину `task_in_quarantine` там выдать невозможно. Зато обратное верно и полезно: на момент связывания `rated_eligible` уже известен — на нём и стоит разделение пулов.
- **Подстраховка на расчёте**: `lock_fighter_ratings` отдаёт `task_in_quarantine` (LEFT JOIN на `battle_tasks`), `battle_runner.settle_battle` дополнительно требует его отсутствия в `should_rate`, причина пишется через новый необязательный параметр `finalize(rated_ineligibility_reason=...)` (через `COALESCE`, чтобы не затереть более раннюю и более конкретную причину). Строковая константа причины живёт в `battle_service`, чтобы словарь причин был в одном месте.
- Подстраховка не должна срабатывать никогда — на это есть отдельный тест: рейтингоспособный бой при пуле, где присутствует карантинная задача, обязан связать НЕ карантинную.
- Новое значение причины добавить в существующий словарь причин на фронте (`lib/api.ts`, `JUDGING_STOP_REASON`/аналог для rated-причин).

## Валидатор `backend/app/services/battle_task_validator.py`
Дешёвые фильтры ДО LLM (бесплатные, отсекают мусор):
1. Длина `prompt` в разумных границах, `rubric` — непустой массив строк, `title` не пустой.
2. Дедуп по `content_key` против существующих не-`rejected` задач.
3. `detect_injection` (переиспользовать из `battle_judges`) по тексту задачи и по каждому пункту рубрики.

LLM-валидация — ОДИН вызов, через `battle_budget.reserve_call` (общий леджер с судьями, дневной кап), критерии отказа:
- нет однозначно проверяемого результата (субъективщина «красиво», «лучше»);
- не self-contained: внешние ссылки, файлы, «актуальные на сегодня» данные, зависимость от времени или случайности;
- двусмысленная формулировка, допускающая несколько несовместимых прочтений;
- нерешаемо за `time_limit_seconds`;
- рубрика не является списком проверяемых критериев;
- заявленная сложность явно не соответствует содержанию;
- запрещённое содержимое.

Возврат: структурированный вердикт `{verdict: accept|reject, reasons: [...], difficulty_assessment: ...}` в `validation_verdict`.

**Что валидатор принципиально НЕ ловит** (записать в докстринг, не делать вид что ловит): знание задачи автором; сговор; семантические инъекции без ключевых слов; неверный эталон, если он правдоподобен; задачу, списанную из закрытого источника.

## API (`backend/app/api/v1/battles.py`)
- `POST /battles/tasks` — авторизованный пользователь. Тело: title, prompt, rubric, category, difficulty. Создаёт `source='user'`, `status='pending_validation'`, запускает валидацию. Квота: не более N заявок в сутки на пользователя (переиспользовать паттерн дневных квот из анти-абьюза).
- `GET /battles/tasks/mine` — свои заявки со статусами и причиной отказа.
- `POST /battles/tasks/{id}/approve` — только админ, `quarantine → ready`, проставляет `approved_by_user_id`, `approved_at`.
- `POST /battles/tasks/{id}/reject` — только админ, с причиной.
- `GET /battles/tasks/moderation` — только админ, очередь на аппрув со статистикой карантина.
- Существующий `POST /battles/tasks/generate` не трогать.

## Фронтенд
- Страница подачи задачи и список своих заявок со статусами. Русская лексика, тёмная тема, Tailwind-утилиты, без новых зависимостей.
- Честно показать: присланные задачи идут в бои **без рейтинга**, пока модератор не одобрит; свою задачу вы в бою не встретите.

## Крайние случаи (нужны тесты)
- Автор — владелец агента A; автор — владелец агента B; автор — владелец обоих. Во всех случаях задача не должна связаться.
- Пул рейтинговых задач пуст, но карантинные есть → рейтинговый вызов не должен получить карантинную задачу.
- Дубль по содержимому с другим регистром и пробелами → отклонён дедупом до LLM.
- Бюджет судей исчерпан → заявка остаётся `pending_validation`, не теряется, не падает 500.
- Инъекция в тексте рубрики, а не в prompt.
- `approve` на задаче в статусе `rejected` → отказ.

## Проверка (условие закрытия)
- `uv run pytest` по battles-тестам — зелёно, включая новые тесты крайних случаев.
- Миграция применяется на живой БД без ошибок.
- Живой прогон через HTTP: подача задачи → валидация → карантин → бой без рейтинга → аппрув → попадание в рейтинговый пул.
- Отдельно доказать исключение автора: автор подаёт задачу, выставляет своего агента, задача не связывается.
